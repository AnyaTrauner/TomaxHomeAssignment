# Fix Notes

Code review prior to the production launch on 2026-03-14. All issues below were found in the original three-file codebase and have been fixed in this commit.

---

| # | File | Issue location (file + row#) | Issue | Why it matters in production | What was changed and why |
|---|------|-------------------------------|-------|------------------------------|--------------------------|
| 1 | `config/database.js` | `config/database.js:2-6` | **Hardcoded database credentials** — `host`, `user`, `password`, and `database` are all literal strings in source code. | The password is visible to anyone with repo access, in every git clone, fork, build log, and Docker image layer. A single leaked repo means full database compromise with no rotation possible without a code deploy. | Replaced all four values with `process.env.DB_*` variables loaded by `dotenv`. Added `.env.example` so the team knows which variables are required, and added `.env` to `.gitignore` so the real values are never committed. |
| 2 | `config/database.js` | `config/database.js:2` | **Single connection instead of a connection pool** — `mysql.createConnection()` creates one TCP connection that is shared by every concurrent request. | A single connection serialises every query. Under any real load, queries queue up and response times spike. Worse, if the connection drops (network blip, DB restart, idle timeout), the entire API goes dark until the process is restarted — there is no automatic reconnect. | Replaced with `mysql.createPool()` with a `connectionLimit` of 10. A pool reconnects automatically, handles concurrency correctly, and is the standard production approach for mysql2. Also switched to the promise-based wrapper (`pool.promise()`) to avoid callback-style async in routes. |
| 3 | `config/database.js` | `config/database.js:8-10` | **Connection error is logged but the process continues** — the callback only calls `console.log` on error, so the app starts up even when it cannot reach the database. | Every incoming request will immediately fail with a cryptic DB error instead of the service refusing to start cleanly. Health checks and orchestrators (Kubernetes, ECS) see the process as healthy when it is not, causing silent failures in production. | Changed to log the error and call `process.exit(1)` so the process fails fast and visibly at startup. This surfaces misconfiguration immediately and allows the orchestrator to restart or alert. |
| 4 | `routes/invoices.js` | `routes/invoices.js:38-41` | **SQL injection via string interpolation** — the INSERT query is built with template literals using raw user input: `` `VALUES ('${clientId}', ${amount}, '${description}', NOW())` ``. | An attacker can send a crafted `clientId` or `description` to read arbitrary data from the database, drop tables, or escalate privileges. `amount` has no quotes around it at all, making numeric injection trivial. This is a critical OWASP Top 10 vulnerability that will be exploited. | Replaced the interpolated query with a parameterised query using `?` placeholders and a values array: `db.query('INSERT INTO invoices ... VALUES (?, ?, ?, NOW())', [clientId, amount, description])`. The mysql2 driver escapes all values — user input never touches the SQL string. |
| 5 | `routes/invoices.js` | `routes/invoices.js:47-48` | **Raw database errors sent to the client** — on any DB error, `res.status(500).send(err)` serialises the full Node.js error object (including the SQL query, table names, and column names) and sends it to the caller. | Internal schema details help attackers enumerate the database structure. It also leaks the original malformed query, which in the old code contained unescaped user input. This violates basic security hygiene and almost certainly violates data protection obligations for a B2B SaaS product. | Removed `res.status(500).send(err)` from the route and replaced with `next(err)`. A centralised error handler in `app.js` logs the full error server-side and returns only a generic `500` message to the client. Sensitive details stay on the server. |
| 6 | `routes/invoices.js` | `routes/invoices.js:12-24` | **No input validation on the POST body** — `clientId`, `amount`, and `description` are used directly without checking whether they exist, have the right type, or have sensible values. | Missing fields produce confusing DB errors (or silent null inserts). An `amount` of `0`, `-1`, or `null` creates corrupt invoice records. A non-numeric `amount` causes an unhandled exception. All of this reaches the database before any error is raised, and the error leaks back to the caller (see issue 5). | Added a `validateInvoiceBody` helper that checks presence, type, and range for all three fields. Invalid payloads get a descriptive `400` response immediately, before any DB call is made. |
| 7 | `routes/invoices.js` | `routes/invoices.js:52-85` | **GET `/:id` is not implemented** — the handler returns `501 Not Implemented` and a plain-text body. | The feature is listed as TODO but the route is registered and will be called by clients. Returning 501 with no documentation will confuse API consumers. More importantly, shipping known-incomplete endpoints to production indicates missing development process gates. | Implemented the handler using a parameterised `SELECT` query. Added an integer validation guard so a non-numeric `:id` gets a `400` rather than a DB error, and a `404` when the invoice does not exist. |
| 8 | `app.js` | `app.js:1-9` (moved to `server.js:7-10`) | **Hardcoded port `3000`** — `app.listen(3000)` gives no flexibility for different environments and no confirmation that the server actually started. | In most container environments the port is injected at runtime via an environment variable. A hardcoded port makes the service impossible to configure without a code change. The lack of a startup log also makes it harder to confirm the process came up correctly. | Changed to `parseInt(process.env.PORT \|\| '3000', 10)` and added a callback to `app.listen` that logs the port. |
| 9 | `app.js` | `app.js:30-37` | **No centralised error handling middleware** — Express has no global error handler, so any `next(err)` call or thrown error in a route results in Express's default HTML error page, which exposes stack traces in development and falls back to an empty response in production. | Stack traces sent to clients reveal file paths, library versions, and code structure — valuable information for an attacker. An empty response to a 500 is also indistinguishable from a network failure for the caller. | Added a four-argument `(err, req, res, next)` error handler at the bottom of `app.js`. It logs the full error internally and always returns a generic JSON `500` body. All routes now use `next(err)` rather than `res.send(err)`. |
| 10 | `app.js` | `app.js:12` | **No security headers** — the API sends no HTTP security headers (no `X-Content-Type-Options`, no `X-Frame-Options`, no `Content-Security-Policy`, etc.). | Missing security headers are a standard finding in any penetration test or compliance audit (SOC 2, ISO 27001). They enable a class of client-side attacks that are trivially mitigated at the server layer. For a B2B SaaS product, this is likely a blocker for enterprise customer security reviews. | Added `helmet()` as the first middleware in `app.js`. Helmet sets a sensible set of security headers with a single line and is the community-standard solution for Express. |
| 11 | `app.js` | `app.js:16-18` | **No request logging** — the application emits no HTTP access logs. | Without access logs there is no visibility into traffic patterns, error rates, slow requests, or potential attacks in production. Debugging any live issue requires guesswork. | Added `morgan` with `'combined'` format in `NODE_ENV=production` (Apache common log format, compatible with most log aggregation tools) and `'dev'` format locally for readable output. |
| 12 | `config/database.js` / `app.js` | `.gitignore:1-3`, `.env.example:1-10` | **No `.gitignore` and no `.env.example`** — the repository has no ignore rules and no documented list of required environment variables. | Without `.gitignore`, a developer will eventually commit a `.env` file with real credentials. Without `.env.example`, the next person to set up the project has no idea what variables are needed. Both are low-effort, high-impact developer-experience and security basics. | Added `.gitignore` (ignores `node_modules/`, `.env`, and log files) and `.env.example` (documents all required variables with placeholder values). |

---

### Dependencies added

| Package | Why |
|---------|-----|
| `dotenv` | Load environment variables from `.env` in development |
| `helmet` | Set HTTP security headers |
| `morgan` | HTTP request logging |

`mysql2` and `express` were already in use; no version changes were made to existing dependencies.
