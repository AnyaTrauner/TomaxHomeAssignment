### Blocking issues
---

| # | Issue location (file + row#) | Issue | Why it matters | What was changed and why |
|---|-------------------------------|-------|------------------------------|--------------------------|
| 1 | `config/database.js:2-6` | **Hardcoded database credentials** â€” `host`, `user`, `password`, and `database` are all strings in code. | The password is visible to anyone with repo access, in every git clone, fork, build log, and Docker image layer. A single leaked repo means full database compromise with no rotation possible without a code deploy. | Replaced all four values with `process.env.DB_*` variables loaded by `dotenv`. Added `.env.example` so the team knows which variables are required, and added `.env` to `.gitignore` so the real values are never committed. |
| 2 | `config/database.js:2` | **Single connection instead of a connection pool** â€” `mysql.createConnection()` creates one TCP connection that is shared by every concurrent request. | A single connection serialises every query. Under any real load, queries queue up and response times spike. Worse, if the connection drops (network blip, DB restart, idle timeout), the entire API goes dark until the process is restarted â€” there is no automatic reconnect. | Replaced with `mysql.createPool()` with a `connectionLimit` of 10. A pool reconnects automatically, handles concurrency correctly, and is the standard production approach for mysql2. Also switched to the promise-based wrapper (`pool.promise()`) to avoid callback-style async in routes. |
| 3 | `config/database.js:8-10` | **Connection error is logged but the process continues** â€” the callback only calls `console.log` on error, so the app starts up even when it cannot reach the database. | Every incoming request will immediately fail with a cryptic DB error instead of the service refusing to start cleanly. Health checks and orchestrators (Kubernetes, ECS) see the process as healthy when it is not, causing silent failures in production. | Changed to log the error and call `process.exit(1)` so the process fails fast and visibly at startup. This surfaces misconfiguration immediately and allows the orchestrator to restart or alert. |
| 4 | `routes/invoices.js:38-41` | **SQL injection via string interpolation** â€” the INSERT query is built with template literals using raw user input: `` `VALUES ('${clientId}', ${amount}, '${description}', NOW())` ``. | An attacker can send a crafted `clientId` or `description` to read arbitrary data from the database, drop tables, or escalate privileges. `amount` has no quotes around it at all, making numeric injection trivial. This is a critical OWASP Top 10 vulnerability that will be exploited. | Replaced the interpolated query with a parameterised query using `?` placeholders and a values array: `db.query('INSERT INTO invoices ... VALUES (?, ?, ?, NOW())', [clientId, amount, description])`. The mysql2 driver escapes all values â€” user input never touches the SQL string. |
| 5 | `routes/invoices.js:47-48` | **Raw database errors sent to the client** â€” on any DB error, `res.status(500).send(err)` serialises the full Node.js error object (including the SQL query, table names, and column names) and sends it to the caller. | Internal schema details help attackers enumerate the database structure. It also leaks the original malformed query, which in the old code contained unescaped user input. This violates basic security hygiene and almost certainly violates data protection obligations for a B2B SaaS product. | Removed `res.status(500).send(err)` from the route and replaced with `next(err)`. A centralised error handler in `app.js` logs the full error server-side and returns only a generic `500` message to the client. Sensitive details stay on the server. |
| 6 | `routes/invoices.js:12-24` | **No input validation on the POST body** â€” request fields are used directly without checking whether they exist, have the right type, or have sensible values. | Missing fields produce confusing DB errors (or silent null inserts). An `amount` of `0`, `-1`, or `null` creates corrupt invoice records. A non-numeric `amount` causes an unhandled exception. All of this reaches the database before any error is raised, and the error leaks back to the caller (see issue 5). | Added a `validateInvoiceBody` helper that checks presence/type/range for `amount` and `description`, and moved client identity to the verified JWT (`req.clientId`) so body `clientId` is ignored. Invalid payloads now get a descriptive `400` response before any DB call. |
| 7 | `routes/invoices.js:52-85` | **GET `/:id` is not implemented** â€” the handler returns `501 Not Implemented` and a plain-text body. | The feature is listed as TODO but the route is registered and will be called by clients. Returning 501 with no documentation will confuse API consumers. More importantly, shipping known-incomplete endpoints to production indicates missing development process gates. | Implemented the handler using a parameterised `SELECT` query. Added an integer validation guard so a non-numeric `:id` gets a `400` rather than a DB error, and a `404` when the invoice does not exist. |
| 8 | `app.js:1-9` (moved to `server.js:7-10`) | **Hardcoded port `3000`** â€” `app.listen(3000)` gives no flexibility for different environments and no confirmation that the server actually started. | In most container environments the port is injected at runtime via an environment variable. A hardcoded port makes the service impossible to configure without a code change. The lack of a startup log also makes it harder to confirm the process came up correctly. | Changed to `parseInt(process.env.PORT \|\| '3000', 10)` and added a callback to `app.listen` that logs the port. |
| 9 | `app.js:30-37` | **No centralised error handling middleware** â€” Express has no global error handler, so any `next(err)` call or thrown error in a route results in Express's default HTML error page, which exposes stack traces in development and falls back to an empty response in production. | Stack traces sent to clients reveal file paths, library versions, and code structure â€” valuable information for an attacker. An empty response to a 500 is also indistinguishable from a network failure for the caller. | Added a four-argument `(err, req, res, next)` error handler at the bottom of `app.js`. It logs the full error internally and always returns a generic JSON `500` body. All routes now use `next(err)` rather than `res.send(err)`. |
| 10 | `app.js:12` | **No security headers** â€” the API sends no HTTP security headers (no `X-Content-Type-Options`, no `X-Frame-Options`, no `Content-Security-Policy`, etc.). | Missing security headers are a standard finding in any penetration test or compliance audit (SOC 2, ISO 27001). They enable a class of client-side attacks that are trivially mitigated at the server layer. For a B2B SaaS product, this is likely a blocker for enterprise customer security reviews. | Added `helmet()` as the first middleware in `app.js`. Helmet sets a sensible set of security headers with a single line and is the community-standard solution for Express. |
| 11 | `app.js:16-18` | **No request logging** â€” the application emits no HTTP access logs. | Without access logs there is no visibility into traffic patterns, error rates, slow requests, or potential attacks in production. Debugging any live issue requires guesswork. | Added `morgan` with `'combined'` format in `NODE_ENV=production` (Apache common log format, compatible with most log aggregation tools) and `'dev'` format locally for readable output. |
| 12 | `.gitignore:1-3`, `.env.example:1-10` | **No `.gitignore` and no `.env.example`** â€” the repository has no ignore rules and no documented list of required environment variables. | Without `.gitignore`, a developer will eventually commit a `.env` file with real credentials. Without `.env.example`, the next person to set up the project has no idea what variables are needed. Both are low-effort, high-impact developer-experience and security basics. | Added `.gitignore` (ignores `node_modules/`, `.env`, and log files) and `.env.example` (documents all required variables with placeholder values). |
| 13 | `server.js:7-20` | **HTTP server could start before DB was ready** â€” the app bound its port immediately, even though DB initialization/connection is asynchronous. | In production this creates a race at startup: readiness checks may pass and traffic can arrive while the DB is still unavailable, causing avoidable 500s and unstable deployments during rollouts/restarts. | Added an async startup function that awaits `db.ready` before calling `app.listen(...)`, and exits fast with a clear startup error if DB readiness fails. This ensures the service only accepts traffic when its critical dependency is actually available. |

## Future suggestions

| Area | Improvement | Why it matters in production | Suggested approach |
|------|-------------|------------------------------|-------------------|
| Stability | Graceful shutdown (`SIGTERM`/`SIGINT`) | Without coordinated shutdown, in-flight requests can be dropped during deploys/restarts, causing partial writes and noisy errors. | In `server.js`, capture the HTTP server instance, stop accepting new connections on signal, await `db.close()`, and exit with a timeout fallback. |
| Stability | Centralized startup config validation | `process.exit(1)` from module import paths can make startup/test behavior brittle and harder to reason about. | Add an explicit `validateConfig()` at bootstrap (`server.js`) for required env vars (`JWT_SECRET`, DB vars by provider) and fail early with actionable messages. |
| Stability | Readiness and liveness endpoints | Orchestrators need health probes that reflect service state; process-up alone is not enough when dependencies are down. | Add `/health/live` (always 200 if process alive) and `/health/ready` (checks DB reachability/readiness promise). |
| Performance | Request body size limits | Large JSON payloads can increase memory pressure and make simple DoS attempts easier. | Replace `express.json()` with `express.json({ limit: '100kb' })` (or team-agreed value) in `app.js`. |
| Performance | DB pool and timeout tuning | Default pool settings may not match production concurrency; long waits under contention increase tail latency. | Make pool settings env-driven (`DB_POOL_LIMIT`, acquire/connect/idle timeouts, keepalive) and tune from load-test results. |
| Performance | Startup warm-up for SQLite path | First request can pay schema/seed initialization cost, increasing p95 latency right after deploy/start. | Trigger DB initialization during startup before listening (already partially done via `db.ready`); ensure init logs/metrics are emitted. |
| Observability | Structured logs + request correlation ID | Plain text logs make cross-service debugging and incident response slower. | Use JSON logs and attach/request-propagate an `X-Request-Id`; include method, path, status, latency, and clientId (when available). |
| Resilience | Transient DB error retry strategy | Brief network hiccups can produce avoidable user-facing failures if every transient error fails immediately. | Add bounded retries with exponential backoff only for known transient classes (connection reset/timeout), never for validation/business errors. |
| Security | Rate limiting on public routes | Even with auth, burst traffic or abuse can starve resources and degrade service for legitimate tenants. | Add route-level rate limiting middleware with conservative defaults and per-client keying, then tune based on production telemetry. |
| Data integrity | Currency precision handling | JavaScript `number` can introduce floating-point rounding issues for money values. | Store monetary values as integer minor units (cents) or use decimal type/library; validate and convert at API boundary. |
